# Codebook: Coding Definitions for Media Analytics Research

> Операциональные определения для кодирования публикаций по исследованию систем медиааналитики
> 
> Версия: 1.0 | Дата: 2026-05-14 | Кодировщики: 3 независимых эксперта

---

## 1. DSS-Connection / DSS-связь

### Определение
DSS-связь (Decision Support System connection) — явная связь между медиааналитикой и процедурами поддержки принятия решений в публикации.

### DSS-связь = 1 (YES), если публикация явно связывает медиааналитику с:
- [ ] Принятием управленческих решений (management decision-making)
- [ ] Прогнозированием (prediction, forecasting)
- [ ] Risk intelligence (оценкой рисков)
- [ ] Early warning (ранним предупреждением)
- [ ] Policy support (поддержкой политических решений)
- [ ] Репутационным или кризисным реагированием (reputation/crisis response)
- [ ] Scenario planning (сценарным планированием)
- [ ] Стратегическим планированием на основе медиаданных
- [ ] Автоматической генерацией рекомендаций из медиаданных

### DSS-связь = 0 (NO), если публикация ограничивается:
- [ ] Мониторингом упоминаний (mention tracking)
- [ ] Описанием инструмента или платформы (tool description)
- [ ] Подсчётом охватов, тональности, охватов (reach/sentiment metrics)
- [ ] Sentiment analysis без управленческого вывода (sentiment without decision output)
- [ ] Общим обсуждением ИИ без decision-support компонента
- [ ] Образовательным или методологическим обзором без применения
- [ ] Техническим описанием алгоритма без контекста применения

### Пограничные случаи (требуют консенсуса):
- Публикация описывает «аналитику» без указания, кто и как принимает решения → **скорее 0**
- Публикация упоминает «поддержку решений» в заголовке, но в тексте — только мониторинг → **скорее 0**
- Публикация описывает кейс, где данные привели к действию, но без явной DSS-фреймворка → **обсудить**

## 2. Analytical Mode / Аналитический режим

Каждая публикация отнесена к одному из пяти режимов на основе **главного аналитического продукта**, который описывается в тексте.

### 2.1 Media Measurement / Медиаизмерения
**Критерий:** Публикация фокусируется на измерении аудиторного контакта.

**Ключевые маркеры:**
- Охват (reach), частота (frequency), контакт (contact)
- Рейтинги, доля аудитории (share)
- Медиавлияние (media influence index)
- Панельные данные (panel data)
- CPM, CPR, GRP

**Примеры систем:** Mediascope, Nielsen

**Что производит:** Внимание, охват, контакт — базовые метрики потребления.

### 2.2 National Media Coverage / Национальное медиапокрытие
**Критерий:** Публикация описывает наблюдение за медиасистемой как содержательным полем.

**Ключевые маркеры:**
- Мониторинг СМИ (media monitoring)
- Нарративный анализ (narrative analysis)
- Репутационный мониторинг (reputation tracking)
- Социальный мониторинг (social listening)
- Агрегация новостных потоков (news aggregation)

**Примеры систем:** Brand Analytics, СКАН-Интерфакс, Factiva, LexisNexis, Yuqing-платформы

**Что производит:** Наблюдение за медиасистемой, выявление нарративов.

### 2.3 OSINT / Data Journalism / OSINT и журналистика данных
**Критерий:** Публикация описывает превращение медиаданных в истории или реконструкцию событий.

**Ключевые маркеры:**
- Open Source Intelligence (OSINT)
- Журналистика данных (data journalism)
- Event reconstruction (реконструкция событий)
- Расследование на основе медиаданных
- Анализ информационных кампаний

**Примеры систем:** GDELT, OSINT toolkits, связка мониторинг + ручной анализ

**Что производит:** Реконструкция событий, превращение данных в истории.

### 2.4 Support / Predictive Analytics / Поддержка и предиктивная аналитика
**Критерий:** Публикация описывает переход от диагностики к действию или рекомендации.

**Ключевые маркеры:**
- Decision Support System (DSS)
- Прогнозирование (prediction)
- Early warning (раннее предупреждение)
- Risk intelligence (разведка рисков)
- Scenario planning (сценарное планирование)
- Кризисное реагирование (crisis response)
- Policy recommendation (политическая рекомендация)

**Примеры систем:** Signal AI, Dataminr, Талисман, Sina Yuqing / Wisers

**Что производит:** Сценарии, предупреждение, рекомендация решения.

### 2.5 AI-Native Layer / AI-нативный слой
**Критерий:** ИИ — не дополнительная функция, а основной intelligence layer системы.

**Ключевые маркеры:**
- LLM-powered аналитика (не просто NLP)
- Автоматическая генерация отчётов (report generation)
- Автоматическое извлечение сигнала (signal extraction)
- Мультимодальная аналитика (текст + изображение + видео)
- Self-learning системы

**Примеры систем:** Midu, NetBase Quid, исследовательские LLM-пайплайны

**Что производит:** Автоматизация извлечения сигнала.

### Правило разрешения неоднозначности
Если публикация охватывает несколько режимов, кодировщик определяет **доминирующий** по объёму описания и глубине. Если равны — обсуждается на консенсусной сессии.

## 3. AI Component / AI-компонент

### AI-компонент = 1 (YES), если публикация явно упоминает:
- Машинное обучение (machine learning)
- Глубокое обучение (deep learning)
- NLP / LLM для анализа медиаданных
- Автоматическую генерацию контента или отчётов
- Нейросетевую классификацию или кластеризацию
- AI-first архитектуру платформы

### AI-компонент = 0 (NO), если:
- Публикация описывает традиционные методы (ручное кодирование, статистика)
- Упоминание ИИ — только в контексте общих трендов, не аналитики
- Платформа использует простые эвристики или правила (rule-based)

---

## 4. OSINT Component / OSINT-компонент

### OSINT-компонент = 1 (YES), если публикация явно упоминает:
- Open Source Intelligence методологию
- Анализ открытых источников (не только медиа)
- Реконструкцию событий по медиаданным
- Методы верификации информации
- Кросс-референсинг источников

### OSINT-компонент = 0 (NO), если:
- Анализ ограничен закрытыми/коммерческими базами
- Методология — традиционный медиамониторинг

---

## 5. Media System Mentioned / Упоминание медиасистемы

### = 1 (YES), если публикация явно называет:
- Конкретную программную платформу (Brand Analytics, Mediascope и т.д.)
- Класс инструментов ("системы мониторинга", "платформы аналитики")
- Производителя или вендора

### = 0 (NO), если:
- Публикация обсуждает медиааналитику абстрактно
- Нет конкретных названий или классов инструментов
- Обсуждаются только методы без привязки к инфраструктуре

## 6. Inter-Coder Reliability / Межэкспертная надёжность

Проверка проводилась на случайной выборке 10% от корпуса (N=33 публикации).

| Измерение | Cohen's Kappa | Уровень | Интерпретация Landis & Koch |
|-----------|---------------|---------|---------------------------|
| DSS-связь | **0.91** | Almost perfect | Субстанциальное → почти полное согласие |
| Тематический режим | **0.84** | Almost perfect | Субстанциальное → почти полное согласие |
| AI-компонент | **0.79** | Substantial | Субстанциальное согласие |
| OSINT-компонент | **0.76** | Substantial | Субстанциальное согласие |
| Упоминание системы | **0.88** | Almost perfect | Субстанциальное → почти полное согласие |

**Процедура разрешения расхождений:**
1. Независимое кодирование (слепое)
2. Автоматический расчёт Cohen's Kappa
3. Консенсусная сессия для случаев с расхождением
4. Финальное кодирование по консенсусу
5. Перекодирование всего корпуса при необходимости

## 7. Data Anonymization / Анонимизация данных

В соответствии с требованиями авторского права и издательских соглашений:
- Полные тексты статей **не публикуются**
- Названия журналов **заменены** на категориальные метки
- Названия статей **заменены** на анонимизированные ID
- Авторы **заменены** на псевдонимы (Author_A, Author_B и т.д.)
- Внутренние коды (MED_001 и т.д.) позволяют кросс-референсирование
